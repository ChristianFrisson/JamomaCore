Introduction

The Tap.Tools Blue DSP Library is a set of open source, cross-platform, C++ objects for performing operations on vectors of audio samples. It the basis of commercial products such as Tap.Tools Max 2.0, Jade 2.0, and Hipno.

 

Objects

Tap.Tools Blue objects are fullly-qualified C++ objects composed of both data elements and algorithms that operate on those elements. Most objects are analogous to a module in a modular systhesizer such as the Moog 900 series. In other words, an object may be an amplifier, an oscillator, a filter, etc. All Tap.Tools Blue objects derive from a common base class, taptools_base, which defines a variety of common macros, shared features, and the basic structure of derived classes.

All objects have attributes, which are the parameters that the user of an object interfaces with. All attributes are accessed via a unified attr_set() and attr_get() mechanism.

All objects perform their algorithm in blocks (or vectors) of samples rather than one sample at a time. This greatly increases the efficiency of the DSP routines. This routine is accessed via a common dsp_vector_calc() method. If an object needs to define multiple dsp methods that have the same input and output configuration, they may defined privately and pointed to by a function pointer called dsp_vector_calc(). If there are varying numbers of inputs or outputs for the different dsp_vector_calc() routines, then you may simply overload the function.

To use a Tap.Tools Blue object, you will need to include the header for that object in the callling code. You will not need to include any additional code in your project. One of the features of C++ is that all code that is included in the class definition header is automatically inlined, without need for additional keywords or options. Tap.Tools Blue objects take advantage of this, which also simplifies file management and reduces the time required to create a new object by integrating the interface and implementation into one entity.

You will also need to include the header for the taptools_audio_signal object. Audio signals are objects that allocate or reference vectors of samples. Working with audio signal objects are detailed elsewhere in this documentation.

 

Instance Creation & Deletion

 

Tap.Tools Blue objects use the standard C++ constructor and destructor mechanism. This means that a user creates an instance of a Tap.Tools Blue object using the new keyword, such as in this example:

tap_gain *my_object; // pointer to a tap_gain object

my_object = new tap_gain();

 

When this function is called, the constructor method of the Tap.Tools Blue object is called. In this example, that object is the tap_gain object. The constructor method is responsible both for allocating any memory, member Tap.Tools Blue objects, and for initializing attributes. The constructor method looks like this:

tap_gain(void) // Constructor

{

    gain = 1.0;

}

 

Likewise, the user will be expected to free the object using the standard free keyword, as in the following example.

delete my_object;

 

In the case of the tap_gain object, there is no memory to free. The destructor function may then be left unimplemented, or an empty destructor may be implemented. Below is the empty destructor method found in the tap_gain object.

~tap_gain(void) // Destructor

{

    ;

}

 

Accessor Methods

 

Accessor methods provide mechanism for setting and querying attribute values in a Tap.Tools Blue object. Typically all attribute values will be declared as private members of a class, thus making it necessary to use an accessor to access those values. An object will typically implement at least two accessor methods: a set method and a get method. The example below shows code that calls both methods of a a member object.

float temp;

my_object->set_attr(tap_gain::k_gain, -6.0);

temp = my_object->get_attr(tap_gain::k_gain); //temp = -6 dB

 

Notice that the first argument is always a selector. Each object defines a set of selectors for choosing which attribute you are accessing with the above methods. For example, inside the tap_gain object the following is the definition of the selectors.

enum selectors{ // Attribute Selectors

    k_gain,

    k_gain_direct,

};

 

The associated routines inside of the tap_gain object are defined in the code fragment that follows.

// Set Attributes

void set_attr(tt_selector sel, tt_attribute_value val) {

    switch (sel){

        case k_gain:

            gain = decibels_to_amplitude(val);

            break;

        case k_gain_direct:

            gain = val;

            break;

    }

}

 

// Get Attributes

tt_attribute_value get_attr(tt_selector sel) {

    switch (sel){

        case k_gain:

            return amplitude_to_decibels(gain);

        case k_gain_direct:

            return gain;

    }

    return 0.0;                    

}

                

DSP Methods

The dsp methods are where the actual work of calculating and processing occurs.  At the moment, this is carried out via a method (or methods) named dsp_vector_calc(). This object will be passed one or more audio signal objects, which contain the input and/or output vectors that will be read or written by the routine.  

For example, the tap_gain class simply scales a signal.  It requires one input signal and one output signal.  In context, we would call the dsp routine as in the following code fragment.

tt_audio_signal *input;

tt_audio_signal *output;

input = new tt_audio_signal(64);

output = new tt_audio_signal(64);

 

// A block of 64 samples are written to the input signal here!

my_object->dsp_vector_calc(input, output);

 

// now use the 64 samples in the output signal to do something.

 

The processing is then performed in the tap_gain class.  The code for that method provides a simple example of how this functions.

void dsp_vector_calc(tt_audio_signal *in, tt_audio_signal *out)

{

    temp_vs = in->vectorsize;

    

    while(temp_vs--)

        *out->vector++ = *in->vector++ * gain;

 

    in->reset(); out->reset();

}

 

Note that this object looks at the first audio signal to determine the number of samples to process as a block.  You must ensure that both signals are of the same block size to guarantee proper operation.  Tap.Tools Blue objects also maintain a vectorsize attribute internally, though this is not used in all objects at this time.  This is likely to change - to be safe, make sure you set the vectorsize of the object and the signals to all match.  

Also note that we are resetting the signals when we are done.  The purpose of this is to make sure that we leave the signals as we found them - meaning that the internal pointer “vector” is returned to the first sample of the block of samples the signal contains.

 

Putting It All Together

Tap.Tools Blue objects are fullly-qualified C++ objects following a structure that is both easy to interface with and is self-documenting in nature.  Please consult the reference portion of this documentation for details on using objects in your code.  To create your own objects, you are encouraged to look at the existing object code.  If you are inclined, please consider submitting your new objects or example code to the project so that it may keep growing!
